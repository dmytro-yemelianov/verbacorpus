# verba — SP15: On-site blog (/blog) + launch articles + sitemap

**Date:** 2026-06-24
**Status:** Approved (design)
**Sub-project:** 15 — a Markdown-sourced `/blog` on verbacorpus.org with the 3 Ukrainian launch articles, plus a `sitemap.xml` (marketing quick-win)
**Repo:** `dmytro-yemelianov/verbacorpus`; `app/`
**Depends on:** the shared chrome (`/chrome.js` topbar + theme), the styles, the i18n catalogs + the Worker HTMLRewriter, the `/p/:id` pages + `/card/:id.png` cards + `/s/<n>` short links (link targets from articles).

---

## 1. Scope & motivation

Give verba an owned, SEO-compounding content home: a `/blog` with the 3 cornerstone Ukrainian articles from the marketing plan (§6), rendered from Markdown at build time. Owned content + the per-proverb pages + a sitemap is the organic-discovery engine. **No corpus data change → no VERSION bump.**

**In scope:** the Markdown→HTML blog build, the 3 articles (Ukrainian), the `/blog` index, a `Блог` nav link, the `sitemap.xml`. **Out of scope:** comments, RSS (nice-to-have, later), translating article bodies (uk only for now — chrome is i18n'd), a CMS.

## 2. Authoring & build

- **Source:** `app/content/blog/<slug>.md`, each with YAML-ish frontmatter:
  ```
  ---
  title: Що таке verba
  slug: shcho-take-verba
  date: 2026-06-24
  lede: Короткий опис для прев'ю та <meta description>.
  ---
  <Markdown body…>
  ```
- **Renderer:** `app/build_blog.py` (Python, build-time dep `markdown`): parse frontmatter + render the body to HTML; wrap each in a **shared page template** (the same `<head>` pattern as about.html — OG/Twitter, canonical, theme-color, the theme-init inline script, spectral + styles.css; the `.topbar` nav incl. the `Блог` link; `<main class="article">` with the rendered body; the footer; `<script type="module" src="/chrome.js">`). Write `app/public/blog/<slug>.html`. Build `app/public/blog/index.html` — the article list (title + date + lede, newest first, linking each slug).
- **Invocation:** add `build_blog.py` to the build sequence (run from `app/`, e.g. `../.venv/bin/python build_blog.py content/blog public/blog`). Document it in the README build steps.
- **Articles link** to proverbs via `/s/<n>` (short links) or `/p/p00NNNN`, and may embed a card with `![](/card/p00NNNN.png)`. Plain Markdown — no custom shortcode (YAGNI).
- **Markdown safety:** the article Markdown is first-party (authored in-repo), so rendered HTML is trusted; still, the renderer escapes nothing beyond what `markdown` does by default (no raw user input).

## 3. Routing & i18n

- `/blog` and `/blog/<slug>` are static assets (served by ASSETS). Add `/blog`, `/blog/*` to `wrangler.jsonc` `run_worker_first` so the Worker's HTMLRewriter **translates the data-i18n chrome** (nav, footer labels) for non-uk routes (`/de/blog`, …). **Article bodies stay Ukrainian** (not translated); only the surrounding chrome is i18n'd — consistent with the "chrome i18n, content uk" model.
- Add `nav.blog` to all 10 catalogs (uk «Блог», en "Blog", de "Blog", fr "Blog", es "Blog", pl "Blog", it "Blog", pt "Blog", ja "ブログ", zh "博客") — the i18n-complete test enforces parity.
- The `Блог` link goes into the shared `.topbar` markup (index.html + about.html + api.html + the blog pages + buildProverbPage's topbar) so the nav is consistent everywhere.

## 4. The 3 launch articles (Ukrainian)

1. `shcho-take-verba.md` — **«Що таке verba»**: what the corpus is, the five sources + counts, how to search (тема/джерело/за змістом), the API + downloads + license. The anchor piece. Links to `/`, `/api.html`, `/about`, a few example `/s/<n>`.
2. `dzherela.md` — **«П'ять зібрань: звідки прислів'я»**: the story of Франко (1901), Номис (1864), Ількевич (1841), Бобкова (1961), Млодзинський (2009), with vivid example proverbs (each linking `/p` + embedding a card), and an honest note on OCR + AI-enrichment limits.
3. `vidkryti-dani.md` — **«Відкриті дані: як користуватися корпусом»**: for researchers/developers — the REST API + formats, Croissant, citing verba (BibTeX/CSL-JSON, and the planned DOI), reuse ideas. Establishes academic credibility.

Each: real content, accurate (no fabrication — same reputation guardrail as the corpus), proper frontmatter, internal links, 1–2 embedded cards where it helps.

## 5. sitemap.xml (marketing quick-win, folded in)

- Generate `app/public/sitemap.xml` listing the stable URLs: `/`, `/about`, `/api.html`, `/blog`, each `/blog/<slug>`, and (optionally, capped) a sample of `/p/<id>` — but 48k `/p` URLs is large; **list the chrome pages + the blog + the language roots; do NOT enumerate all 48k proverbs** (avoid a giant sitemap; the `/p` pages are reachable + indexable via internal links + the API). If a proverb sitemap is wanted later, generate a separate paginated `sitemap-proverbs.xml`. Reference `sitemap.xml` from `robots.txt`.
- Generated by `build_blog.py` (or a tiny step) so the blog slugs are always included.

## 9. Risks / open items

- **Markdown dep in the build:** `markdown` is build-time only (Python, in `.venv`), not bundled into the Worker — no runtime impact. Pin it in a build requirements note.
- **Article accuracy:** the articles describe the corpus + sources — must match reality (counts, source editions from SP13, license). No overclaiming (marketing guardrail).
- **Sitemap size:** deliberately NOT enumerating 48k `/p` URLs (would bloat); chrome + blog + lang roots only. Documented.
- **Chrome consistency:** the `Блог` link must be added to ALL topbar instances (index/about/api/blog/`/p`) or the nav diverges per page.
